# BLOCKERS_AND_ALIGNMENT_POLICY.md

## 文件用途

本文件定义 AI Agent 在开发过程中的“阻塞停顿与需求对齐”规则。

本项目不允许 AI Agent 在关键不明确、工具受限、权限受限、外部依赖缺失或可能影响最终交付质量的情况下强行猜测并继续实现。

本文件适用于：

- Codex
- Claude Code
- Cursor
- 其他 AI Coding Agent

---

## 总原则

AI Agent 可以自主完成明确、低风险、可逆、不会改变核心架构方向的常规开发任务。

但遇到以下情况时，必须停止继续推进当前相关实现，向用户说明阻塞点，并等待用户确认、补充信息、提供资源或帮助完成外部操作。

不要为了“看起来完成了”而编造、猜测、伪造验证结果或生成不可维护的半成品。

---

## 必须停下来问用户的情况

### 1. 需求不明确且会影响核心架构

例如：

- 用户 / 租户 / 平台方的权限边界不明确
- 租户计费、平台计费、用户计费规则不明确
- Provider / Endpoint / Protocol 的关系与文档冲突
- ModelRoute 路由优先级不明确
- API Key 权限范围不明确
- Chat / Playground / Endpoint 测试调用链不明确
- 前后端仓库和 Git Submodule 实际远程地址不明确
- 第一阶段和完整蓝图的边界不明确

处理方式：

```text
停止相关实现 -> 列出不明确点 -> 给出可选方案 -> 请求用户确认
```

### 2. 会影响数据模型或后续重构成本

例如：

- 数据库核心表字段无法确定
- tenant_id / owner_tenant_id / owner_user_id 归属不明确
- 账单字段含义不明确
- 事件幂等字段不明确
- PostgreSQL JSONB / numeric / 索引设计会影响长期扩展
- 是否需要保留某个重要扩展点不明确

处理方式：

```text
不要强行定死设计，必须询问用户。
```

### 3. 会影响安全、资金、计费或权限

例如：

- API Key 生成、展示、存储规则不明确
- Provider API Key 加密方案涉及真实密钥
- 真实支付、充值、余额、扣费逻辑不明确
- 是否记录 Prompt / Response 审计不明确
- 管理员是否能查看租户数据不明确
- 跨租户资源访问规则不明确

处理方式：

```text
必须停下来确认，不能猜。
```

### 4. 涉及真实外部资源或凭证

例如：

- GitHub / GitLab 真实仓库地址
- 前端 Git Submodule 远程 URL
- Figma 链接或 token
- OpenAI / Anthropic / OpenRouter / DeepSeek API Key
- PostgreSQL / Redis 生产连接
- Docker Registry 凭证
- 云服务器、Kubernetes 集群、域名、证书

处理方式：

```text
不能编造，不能写假值冒充真实配置。
必须使用占位符，并请求用户提供真实值或手动完成对应操作。
```

### 5. 工具、Skill、MCP、插件不可用

例如：

- 无法使用 superpowers skill
- 无法使用 impeccable skill
- 无法使用 taste skill
- 无法使用 Context7 MCP
- 无法使用 Playwright MCP
- 无法使用 PostgreSQL / Redis MCP
- 无法访问浏览器或本地服务
- 无法执行 docker compose
- 无法执行 git submodule

处理方式：

```text
如果该工具不是当前步骤的强依赖，可以继续实现，并记录未验证项。
如果该工具是当前步骤验证质量的必要条件，必须停下来说明：
1. 缺少什么
2. 影响什么
3. 用户需要怎么帮助
4. 可以先做哪些不依赖该工具的部分
```

禁止：

- 声称使用了实际未使用的工具
- 声称验证通过但实际未验证
- 因工具缺失直接跳过关键验证且不告知用户

### 6. 命令失败且无法自行修复

例如：

- 项目无法编译
- Docker Compose 启动失败
- PostgreSQL 迁移失败
- Redis 连接失败
- 前端 npm install / build 失败
- Playwright 无法打开页面
- Maven / Gradle 依赖无法解析
- Git Submodule 添加失败

处理方式：

```text
先尝试合理修复。
如果仍失败，停止并提供：
1. 已执行命令
2. 失败日志摘要
3. 已尝试的修复
4. 需要用户提供的信息或执行的操作
5. 不影响当前问题的后续可做事项
```

### 7. 需要用户进行外部操作

例如：

- 创建远程仓库
- 创建前端仓库
- 添加 Git Submodule 真实 URL
- 配置 API Key
- 授权 MCP
- 登录 GitHub / GitLab
- 安装 Docker / Node / JDK
- 开启端口
- 启动本地数据库
- 提供 Figma 权限

处理方式：

```text
必须停下来，让用户完成或提供结果。
不能假装已经完成。
```

### 8. 前端 UI 技术选型无法完成有效评估

前端 UI 框架、组件库、样式方案、图表库、表格方案、Markdown 渲染、代码高亮、主题方案不能擅自决定。

如果无法使用：

- impeccable skill
- taste skill
- Context7 MCP
- Playwright MCP

则必须说明评估受限，并询问用户是否：

1. 先按保守可维护方案推进
2. 等用户安装工具后再选型
3. 用户指定 UI 技术栈

不得直接拍脑袋选一个框架。

## 可以自主推进的情况

以下情况可以继续，不必反复问用户：

- 明确的代码结构整理
- 根据已确认文档创建目录
- 编写公共 DTO / enum / model
- 编写单元测试
- 修复明显编译错误
- 添加 README / 文档说明
- 实现已经明确的接口骨架
- 使用占位符替代未知真实密钥
- 对非核心功能保留扩展点
- 根据已有约束实现 PostgreSQL / Redis 本地开发配置

## 提问格式要求

当必须停下来时，AI Agent 应使用以下格式：

```md
## 需要你确认 / 协助的阻塞点

### 阻塞点 1：<标题>
- 当前情况：
- 为什么会影响交付质量：
- 我已经尝试：
- 可选方案：
  1. ...
  2. ...
  3. ...
- 我建议：
- 需要你提供 / 确认：

### 我可以先继续做的不受影响事项
- ...
```

## 禁止行为

禁止：

- 不明确时擅自决定核心业务规则
- 工具不可用时假装已经验证
- 编造仓库地址、API Key、Figma 链接、MCP 可用状态
- 把 mock 当成真实实现
- 为了继续推进而牺牲安全、权限、计费、租户隔离
- 把无法完成的外部操作写成已经完成
- 遇到阻塞只写 TODO 然后继续大量开发
- 把 ai-common 做成第三个服务
- 把前端源码直接提交进后端仓库普通目录
- 未评估就擅自选择 UI 框架

---

## 最终交付要求

最终交付说明必须包含：

- 哪些地方已经完成
- 哪些地方因为阻塞没有完成
- 哪些工具 / MCP / Skill 实际使用过
- 哪些工具 / MCP / Skill 未安装或未确认
- 哪些验证实际执行过
- 哪些验证没有执行及原因
- 哪些地方需要用户后续提供真实配置
- 当前项目是否能编译、启动、运行最小测试
